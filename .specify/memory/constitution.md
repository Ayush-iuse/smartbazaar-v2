# SmartBazaar AI — Constitution V3 (AI-Native Enterprise-Grade)

This constitution governs all product, architecture, development, and quality validation decisions in the SmartBazaar AI V3 project. All contributors (human engineers and AI agents) MUST strictly follow these principles and guidelines.

---

## 1. Core Principles

### Principle 1: Security First
- **Mandatory Validation**: All client inputs MUST undergo structural, type, and bounds checks before any service processing. Pydantic V2 schemas are the single source of truth for parameter boundaries.
- **No Raw SQL**: Database queries MUST use SQLAlchemy ORM parameters exclusively. Hand-crafted raw SQL queries or string interpolations inside SQL strings are strictly forbidden.
- **Environment Isolation**: JWT secret keys, API credentials, and database connection strings MUST be loaded dynamically from environment variables (`.env` or system variables) via config parsing classes. No secrets may be hardcoded.
- **Secure Hashing**: Passwords MUST be salted and hashed using `bcrypt` prior to database persistence.
- **Authentication Rate Limiting**: All login and register endpoints MUST implement request throttling rules to mitigate brute-force vector attacks.
- **No Implementation Leaks**: API responses must mask system stack traces, database schemas, and folder layouts. Outgoing responses must utilize strict response models.
- **Prompt Sanitization**: All user-provided fields passed to Large Language Models (LLMs) MUST be sanitized to block prompt injection attacks.

### Principle 2: Explainable AI
- **Explainability Payload**: Every endpoint or module delivering an AI-generated suggestion MUST return a response payload containing:
  1. `recommendation`: The core value suggested (e.g. price range, category, description text).
  2. `confidence`: A percentage confidence score (0-100%) indicating recommendation reliability.
  3. `explanation`: A clear, natural language explanation detailing why the recommendation was selected.
- **No Black Boxes**: Raw or unexplained numbers/labels generated by models must not be exposed to the client without supporting explainability attributes.

### Principle 3: AI Transparency
- **Explicit Labeling**: All AI-generated elements visible on the UI must carry the badge: `"AI Generated Suggestion"`.
- **User Agency**: AI recommendations are advisories only. The user MUST retain final edit control over all text, valuations, categories, and tags.
- **No Automated Side-Effects**: No system mutations (e.g. publishing listings, sending messages, deleting profiles) may occur automatically from AI suggestions without the user's explicit manual confirmation (e.g. clicking "Approve" or "Save").

### Principle 4: Agent Driven Architecture
- **Agent Mapping**: Every workflow in the SDLC must map to a specific AI Agent context. All changes are verified against the agent's defined scope, inputs, and outputs.

### Principle 5: Marketplace Intelligence
- **Active Insights**: The platform must deliver real-time intelligence reports alongside raw data items, calculating metrics such as:
  - *Competition Score*: Indicating listing density and pricing competition in the category.
  - *Sale Probability*: Modeling the likelihood of a sale based on price, category, and condition.
  - *Listing Quality Score*: Evaluating titles, descriptions, and pictures for completeness.
  - *Market, Category, and Fraud Trends*: Exposing historical demand shifts and flagging active scams.

### Principle 6: User Experience
- **Effort Reduction**: Workflows must minimize manual keyboard input. Recommend autocompletes, defaults, and single-click choices.
- **Accessibility Integration**: Interactive elements must follow WCAG AA guidelines (keyboard-navigable focus, explicit colors contrast thresholds, text screen-reader labels, and image alt attributes).
- **Mobile-First Layout**: All screens must be responsive, touch-friendly, and optimized for phone resolutions.

### Principle 7: Theme Consistency
- **Visual Options**: System must support Light, Dark, and System-Preference color themes.
- **Preference Persistence**: The user's chosen theme must persist across browser reloads (e.g. via local storage sync) and prevent page-load flickering.

### Principle 8: AI Native SDLC
- **Rigid Lifecycle Flow**: Features must follow the lifecycle path: `Constitution` → `Specify` (PRD) → `Plan` (Architecture) → `Tasks` (Checklist) → `Implement` → `Validate` → `Deploy`.
- **Pre-requisite Artifacts**: No code edits may occur without a verified PRD file (`spec.md`), an implementation architecture document (`plan.md`), and an active checklist (`tasks.md`).

### Principle 9: Internship Scope
- **Time Boxing**: Features must be designed to be complete within 7-10 days by a single developer.
- **Zero Cloud Dependencies**: The application runtime must compile, build, and run entirely in local development environments (e.g. SQLite databases, Docker Compose local networking) with zero dependency on active paid cloud service subscriptions.

### Principle 10: Working Software First
- **MVP Validation**: A working end-to-end MVP utilizing simple rules-based fallbacks is preferred over highly complex, unstable AI models.
- **Graceful Degradation**: If an AI API endpoint is unavailable, the fallback algorithm must trigger instantly, yielding a default suggestion labeled with an offline warning badge.

## 2. Agent Definitions

### 1. Product Manager Agent
- **Responsibilities**: Gathers product requirements, designs user scenarios, defines acceptance criteria, and manages feature scope.
- **Inputs**: User requirements, system constraints, user feedback.
- **Outputs**: `spec.md` (Product Requirements Document), User Stories, and Acceptance Tests.
- **Success Metrics**: 100% of user stories map to verifiable code routes; specifications require zero major rewrites during architecture phases.

### 2. Architect Agent
- **Responsibilities**: Outlines project system context, defines data layouts, organizes folder hierarchies, and specifies dependencies.
- **Inputs**: `spec.md`, technology stack rules.
- **Outputs**: `plan.md` (Architecture Design), Mermaid system context diagrams, database schema designs.
- **Success Metrics**: Zero circular dependencies between backend services; architecture document passes all security gates.

### 3. Backend Agent
- **Responsibilities**: Implements FastAPI routers, Pydantic schemas, database models, and server-side logic.
- **Inputs**: `plan.md`, Swagger API contracts.
- **Outputs**: Clean Python codebase, SQLite database integrations, API endpoints.
- **Success Metrics**: 100% compliance with defined schemas; API response latency stays under 200ms for standard CRUD; zero connection leaks.

### 4. Frontend Agent
- **Responsibilities**: Builds Next.js pages, responsive Tailwind CSS layouts, Zustand state stores, and theme selectors.
- **Inputs**: `plan.md`, REST endpoint payloads.
- **Outputs**: Clean TypeScript/Next.js pages and components.
- **Success Metrics**: Zero compile-time warnings; Lighthouse accessibility score above 90; visual responsiveness down to mobile resolutions.

### 5. QA Agent
- **Responsibilities**: Builds automated pytest suites, checks API boundaries, runs lint checks, and conducts manual verification walkthroughs.
- **Inputs**: `spec.md`, source files, pytest contexts.
- **Outputs**: `tests/test_*.py` files, `docs/test-report.md`.
- **Success Metrics**: 100% test pass rates; test coverage covers all P0 routes; zero regressions found in regression checks.

### 6. Security Agent
- **Responsibilities**: Scans for input vulnerabilities, implements rate-limiting blocks, filters prompt inputs, and verifies password hashing scopes.
- **Inputs**: Source files, Docker configurations, config files.
- **Outputs**: `docs/security-review.md`, input validation utilities.
- **Success Metrics**: Zero secrets committed to the repository; zero vulnerability reports on SQL Injection or XSS filters.

### 7. Seller Copilot Agent
- **Responsibilities**: Orchestrates description generation, category lookups, pricing guides, and scans listings for scam indicators.
- **Inputs**: User titles, keywords, item conditions.
- **Outputs**: AI descriptions, price range intervals, fraud risk level details.
- **Success Metrics**: AI responses yield full explainability payloads (recommendation + confidence + explanation); fallback matches under 3 seconds.

### 8. Buyer Agent
- **Responsibilities**: Interfaces search engines, manages filters (location/category), and handles localized chat threads.
- **Inputs**: Search query `q`, message content text.
- **Outputs**: Filtered listing arrays, persistent chat message records.
- **Success Metrics**: Real-time response message mock delays trigger exactly at 2.0 seconds.

### 9. Marketplace Intelligence Agent
- **Responsibilities**: Calculates listing quality scores, category demand values, competition metrics, and sale probability scores.
- **Inputs**: Database listings tables, historical pricing metadata.
- **Outputs**: Marketplace analytics, trends reports, listing score values.
- **Success Metrics**: Intelligence evaluations complete in under 2 seconds.

### 10. Search Agent
- **Responsibilities**: Computes text indexes, filters query arguments, and returns search feeds.
- **Inputs**: Query keyword `q`, location parameters, category filter tags.
- **Outputs**: Ordered listing lists.
- **Success Metrics**: Keyword query search returns results within 200ms.

## 3. Development Rules

1. **No Code Without Tasks**: No development task may be executed unless documented under `tasks.md`.
2. **Strict Folder Cleanliness**: All endpoints must reside under `backend/app/routers/` (no duplicate `routes/` folders). All shared UI components must be modularized under `frontend/src/components/`.
3. **No Direct Mutations**: AI suggestions must write to internal staging states first. Mutating persistent database models requires an explicit user action (e.g. clicking "Save Listing").
4. **Offline Rule Compliance**: All service code calling external APIs (e.g. OpenAI) MUST check for credentials and gracefully fall back to local rule-based helper utilities on failure or key absence.

## 4. Review Checklist

- [ ] Does the PR introduce any raw SQL queries? (Reject if yes)
- [ ] Are all API inputs validated via Pydantic models? (Reject if missing)
- [ ] Do all AI endpoints return a confidence score and a detailed reason? (Reject if missing)
- [ ] Is there an `"AI Generated Suggestion"` transparency badge visible on all AI UI elements? (Reject if missing)
- [ ] Does the UI support light, dark, and system theme settings, and does it persist? (Reject if missing)
- [ ] Do backend integration tests pass cleanly via pytest? (Reject if failing)
- [ ] Does the frontend build cleanly without warnings or errors? (Reject if failing)

---

## 5. Architecture Governance Rules

- **Monolith Architecture Enforcement**: The system must operate as a single-repository monolith separating frontend and backend services. Microservices are prohibited.
- **Docker Compose Standard**: All local services (database, backend, frontend) must orchestrate using a single `docker-compose.yml` file, leveraging multi-stage builder images to keep runner sizes minimal.
- **Schema Control**: The database schema must register dynamically via SQLAlchemy declarative bases. All table schemas must follow strict foreign key mappings and cascade rules.
